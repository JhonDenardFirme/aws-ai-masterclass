The Seven Phases │ ├── 01 Business Goal Identification — define objective, success criteria, constraints │ ├── 02 ML Problem Framing — translate into classification, regression, clustering, etc. │ ├── 03 Data Processing │ │ ├── Data Collection — gather raw data from sources │ │ ├── Data Preprocessing — clean, transform, handle missing values │ │ └── Feature Engineering — extract and transform variables for predictive signal │ ├── 04 Model Development │ │ ├── Training — fit model to labeled data │ │ ├── Tuning — adjust hyperparameters for better performance │ │ └── Evaluation — validate against held-out data │ ├── 05 Model Deployment — integrate into production for real-world predictions │ ├── 06 Model Monitoring — track performance, detect drift, flag degradation │ └── 07 Model Retraining — update with new data when performance drops │ └── ML is an iterative loop — monitoring triggers retraining, NOT a linear process ending at deployment

Business → ML Translation │ ├── Business problem: inefficient call routing — wrong agent, delays, poor experience │ ├── ML task: predict correct agent skill = Multiclass Classification │ └── Learning type: Supervised Learning — historical calls with known routing outcomes as labels │ ├── B2. D. Model Sources and Built-in Algorithms │ ├── D1. Four Source Options (effort → flexibility spectrum) │ ├── Pretrained Models — least effort · ready to deploy or fine-tune · JumpStart / Bedrock │ ├── Built-in Algorithms — moderate effort · SageMaker-native · configure, not code │ ├── Custom Framework Models — high flexibility · TensorFlow, PyTorch, Scikit-learn │ ├── Custom Docker Models — maximum control · any language/library · highest effort │ └── Model source = effort vs flexibility trade-off │ └── D2. E
Evaluation Datasets and Model Fit
E
E. Evaluation Datasets and Model Fit │ ├── E1. Three Dataset Roles │ ├── Training Set — model learns from this · fits parameters · ~70–80% │ ├── Validation Set — guides tuning iteratively · evaluate after each run · ~10–15% │ ├── Test Set — final unbiased evaluation · once-only · ~10–15% │ ├── Test set must remain sealed until all development decisions are made │ └── Data leakage: using test set during development contaminates it as a fair estimate │ ├── E2. Model Fit States │ ├── Underfitting — High Bias · Low Variance · poor on train AND test · model too simple │ ├── Overfitting — Low Bias · High Variance · great train · poor test · memorized noise │ └── Balanced ✓ — Low Bias · Low Variance · generalizes well · target state │ └── E3. Bias-Variance Trade-off ├── Bias — error from wrong assumptions · off-target predictions · underfitting ├── Variance — sensitivity to noise · scattered results · overfitting ├── Bullseye analogy: center = true value · spread = variance · distance = bias └── Ideal: Low Bias + Low Variance = tight cluster at the bullseye
F
Classification Metrics
F
F. Classification Metrics │ ├── F1. Confusion Matrix │ ├── True Positive (TP) — correctly predicted positive │ ├── True Negative (TN) — correctly predicted negative │ ├── False Positive (FP) — wrong alarm · predicted positive, actually negative │ └── False Negative (FN) — missed positive · predicted negative, actually positive │ ├── F2. The Five Metrics │ ├── Accuracy — (TP+TN)/(TP+TN+FP+FN) │ │ └── Misleading with imbalanced data — avoid relying on alone │ ├── Precision — TP/(TP+FP) · minimize false positives │ │ └── Use when FP is costly · example: spam filtering │ ├── Recall (Sensitivity) — TP/(TP+FN) · minimize false negatives │ │ └── Use when FN is critical · example: disease detection │ ├── F1 Score — harmonic mean of Precision and Recall │ │ └── Use when both FP and FN matter · imbalanced data │ └── AUC-ROC — area under ROC curve (TPR vs FPR) │ ├── AUC ≈ 0.5 — random guessing │ ├── AUC = 1.0 — perfect classifier │ └── Threshold-independent — compare models before choosing decision cutoff │ └── Metric choice = business cost of each error type (FP vs FN)
G
Regression Metrics
G
G. Regression Metrics │ └── G1. MSE and R² ├── MSE — Mean Squared Error │ ├── Average of (prediction − actual)² across all samples │ ├── Squaring penalizes large errors more than small ones │ ├── Lower MSE = better model — minimize toward zero │ └── Sensitive to outliers — one large error inflates MSE significantly ├── R² — Coefficient of Determination │ ├── Proportion of target variance explained by the model │ ├── Range 0 to 1 (can be negative for very poor models) │ ├── R² = 1.0 → perfect fit · R² = 0.0 → no better than mean │ └── Unitless — comparable across different scales └── Together: MSE = error magnitude · R² = goodness of fit

Business Metrics, Deployment, and Inference
H
H. Business Metrics, Deployment, and Inference │ ├── H1. Business KPI Alignment │ ├── ML metrics — accuracy, F1, MSE (model behavior in isolation) │ ├── Business metrics — sales, cost, retention (real-world impact) │ ├── High ML metrics ≠ business success — must optimize the right thing │ ├── Error Cost Asymmetry │ │ ├── FP → operational cost (manual reviews, customer friction) │ │ └── FN → missed revenue, safety risk │ ├── Cost Function — assign dollar values to errors · optimize threshold for business │ ├── A/B Testing — two model versions live simultaneously · compare business outcomes │ └── Canary Deployment — new model receives small % of traffic before full rollout │ ├── H2. Deployment Types │ ├── Self-Hosted API │ │ ├── Full infrastructure control · customization flexibility │ │ └── High operational overhead · manual scaling and maintenance │ └── Managed API (Amazon SageMaker) │ ├── No infra management · auto scaling · secure HTTPS · multi-model hosting │ └── Less control than self-hosted │ └── H3. Four Inference Types ├── Real-time Inference │ ├── Low latency · persistent endpoint · interactive applications │ └── Always-on cost ├── Batch Transform │ ├── Large dataset · no persistent endpoint · offline scoring │ └── Lower cost · higher latency acceptable ├── Asynchronous Inference │ ├── Large payloads · long processing time · queued requests │ └── Use for video analysis, long document processing └── Serverless Inference ├── No infra management · auto-scale · best for intermittent traffic └── Cold start latency trade-off

Definition and Need │ ├── MLOps = DevOps for ML — people + processes + technology │ ├── Scope: development → deployment → monitoring → retraining │ ├── Models degrade: data drift (feature distribution shifts) │ ├── Models degrade: concept drift (feature→label relationship changes) │ └── Goal: keep models useful over time with automated oversight │ ├── I2. Five Benefits │ ├── Productivity — faster experimentation, access to curated datasets │ ├── Reliability — CI/CD improves consistency, reduces human error │ ├── Repeatability — automated pipelines produce reproducible results │ ├── Auditability — track data, models, experiments for compliance │ └── Data & Model Quality — continuous bias and drift monitoring │ ├── I3. Four Core Principles │ ├── Version Control — track code, data, models · enables rollback │ ├── Automation — automate ingestion, training, testing, deployment │ ├── CI/CD for ML — 4 components: │ │ ├── Continuous Integration — test code, data, models │ │ ├── Continuous Delivery — deploy models automatically │ │ ├── Continuous Training — retrain models with new data │ │ └── Continuous Monitoring — track model + business metrics │ └── Model Governance — fairness, compliance, security, documentation │ ├── I4. Two MLOps Pipelines │ ├── Training Pipeline │ │ ├── Data preparation → Feature engineering │ │ ├── Model build → Training run │ │ ├── Evaluation → meets criteria? │ │ └── Registration → Model approval gate │ └── Deployment Pipeline │ ├── Source → Build → Deploy → Test → Promote │ ├── Production monitoring: infra · data quality · model performance │ └── Drift detected → triggers retraining → loops to training pipeline │ └── I5. AWS MLOps Stack ├── SageMaker Pipelines — automate end-to-end ML workflows ├── Model Registry — version, approve/reject model artifacts ├── SageMaker Experiments — track runs, hyperparameters, metrics ├── Model Monitor — detect drift, trigger retraining alerts ├── Feature Store — versioned features for training + inference consistency └── Data Wrangler — low-code prep integrated directly into Pipelines
Module Summary
  • ML lifecycle = 7-phase iterative loop: Business Goal → Problem Framing → Data Processing (Collection · Preprocessing · Feature Engineering) → Model Development → Deployment → Monitoring → Retraining. Loop never ends.
  • Call center routing: multiclass classification + supervised learning. Features: orders, device, membership. Split: 70–80/10–15/10–15. Test set sealed. Never evaluate on training data.
  • SageMaker tools: Data Wrangler · Feature Store (consistency, prevents skew) · Canvas (no-code) · JumpStart (pretrained) · Experiments (tracking) · Hyperparameter Tuning · Model Monitor (drift). Artifacts in S3.
  • Model sources: Pretrained → Built-in → Custom Framework → Custom Docker (effort ↑, flexibility ↑). Algorithm categories: Supervised · Unsupervised · Image · Text. Selection = data type + task type.
  • Model fit: Underfitting (high bias) · Overfitting (high variance) · Balanced (both low). Classification: Accuracy (imbalanced limitation) · Precision (FP cost) · Recall (FN cost) · F1 (balance) · AUC-ROC (threshold-independent). Regression: MSE (error magnitude) + R² (goodness of fit). Inference: Real-time · Batch · Async · Serverless.
  • MLOps = DevOps for ML. 5 benefits (Productivity · Reliability · Repeatability · Auditability · Quality) · 4 principles (Versioning · Automation · CI/CD · Governance) · CI/CD = 4 components (Integration · Delivery · Training · Monitoring). Two pipelines: Training + Deployment. AWS stack: Pipelines · Registry · Experiments · Monitor · Feature Store · Data Wrangler.
