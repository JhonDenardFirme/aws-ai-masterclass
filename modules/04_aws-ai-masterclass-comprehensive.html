Module Introduction

The first three modules established what AI and ML are, where they apply, and how to build them responsibly. Module 4 addresses how to actually develop, evaluate, deploy, and operate an ML system end-to-end — from translating a business problem into an ML task, through choosing the right SageMaker tools, all the way to measuring success with the right metrics and keeping models healthy in production through MLOps.

This is the most technically dense module in the MasterClass. It introduces the full ML lifecycle, every major SageMaker service, the complete set of model evaluation metrics, and the operational discipline of MLOps. Every concept builds on the previous — a model is only as good as its data, its evaluation, and its post-deployment monitoring.

A
The ML Development Lifecycle
A
A1The Core Phases

Building an ML system is not a one-time event — it is a continuous, iterative loop that cycles through the same phases repeatedly as new data arrives, business needs evolve, and model performance changes. Understanding the lifecycle prevents the most common failure mode: treating ML as a one-shot project that ends at deployment.

01
Business Goal Identification
Define the specific business objective — what problem needs solving, what success looks like, and what constraints apply (cost, latency, compliance).
02
ML Problem Framing
Translate the business goal into a concrete ML task. Is this classification, regression, clustering, or something else? What label do we predict? What features are available?
03
Data Processing
Three sub-steps: Data Collection (gather raw data from sources), Data Preprocessing (clean, transform, handle missing values), Feature Engineering (extract, combine, or transform variables to maximize predictive signal).
04
Model Development
Training (fit a model to data), Tuning (adjust hyperparameters for better performance), Evaluation (validate against held-out data).
05
Model Deployment
Integrate the trained model into a production system where it generates real-world predictions for end users or downstream systems.
06
Model Monitoring
Track model performance over time in production — detect data drift, performance degradation, and unexpected behavior.
07
Model Retraining
Update the model with new data when monitoring reveals performance has degraded or the data distribution has shifted.
Critical Exam Point

ML is an iterative loop, not a linear process. Monitoring triggers retraining, retraining feeds new evaluation, evaluation may change deployment — the cycle never ends. This is the fundamental distinction between ML and traditional software development.

Key Points
  • Seven phases: Business Goal → Problem Framing → Data Processing → Model Development → Deployment → Monitoring → Retraining.
  • ML is iterative — monitoring and retraining form a feedback loop, not a termination point.
  • Data Processing has three sub-steps: Collection → Preprocessing → Feature Engineering.
B
Real-World ML Example — Amazon Call Center
B
B1Problem Framing: Business → ML

The call center example illustrates how every step of the lifecycle applies to a real problem. The business problem is inefficient call routing — calls being sent to the wrong agent, causing delays, extra transfers, and poor customer experience. The business goal is to reduce these transfers and improve satisfaction.

Translating to an ML task: if we can predict the correct agent skill before the call is connected, routing becomes automatic and accurate. Since there are multiple possible agent skills (billing, tech support, device help, etc.), this is a multiclass classification problem — predicting one of several categorical outcomes.

The learning type is supervised learning — historical call records (with known correct routing outcomes) serve as labeled training data. The model learns from past examples to predict future routing decisions.

Feature Engineering

Three key features were identified: Recent orders (what the customer recently bought), Device ownership (e.g., whether they own a Kindle), and Membership status (Prime, standard, etc.). These signal why the customer is likely calling before they say a word.

B2Data Preparation and Training

Data preprocessing involves two key steps here: cleaning (removing erroneous or incomplete records) and label refinement — combining similar labels into unified categories. For example, multiple distinct Kindle-related issue labels might be merged into a single "Kindle Support" label, reducing noise and improving model reliability. [+]

The dataset is split into three non-overlapping subsets. A standard split is 70–80% training, 10–15% validation, and 10–15% test. The model trains on the training set, is tuned against the validation set, and is evaluated once — only once — on the test set before deployment.

Never Evaluate on Training Data

A model evaluated on its own training data will always appear better than it actually is — it has already "seen" those examples. Always evaluate on held-out data. The test set must remain untouched until the final evaluation step; using it for tuning invalidates it as a fair measure.

Key Points
  • Call center routing = multiclass classification (multiple possible agent skill classes).
  • Learning type = supervised (historical records with known routing outcomes as labels).
  • Label refinement: merge similar labels to reduce noise (e.g., multiple Kindle issues → one label).
  • Standard data split: 70–80% train · 10–15% validation · 10–15% test.
  • Success metric = business improvement (fewer call transfers), not just model accuracy.

Model Sources and Built-in Algorithms
D
D1The Four Model Source Options

Every ML project involves a choice of where to start from. The options form a spectrum from least effort (pretrained) to maximum control (custom Docker), with corresponding trade-offs in flexibility and required expertise.

Pretrained Models
  • Ready to deploy or fine-tune
  • Least effort — fastest time to value
  • Available via JumpStart or Bedrock
  • Limited customization without fine-tuning
Built-in Algorithms
  • Provided by SageMaker natively
  • Moderate effort — configure, don't code
  • Optimized for SageMaker compute
  • Cover common supervised + unsupervised tasks
Custom Frameworks
  • TensorFlow, PyTorch, Scikit-learn, etc.
  • High flexibility — write your own model code
  • Full algorithm control
  • Runs in SageMaker-managed containers
Custom Docker
  • Fully customized environments
  • Maximum control — any language, any library
  • Highest effort — build and maintain container
  • Required when standard containers won't work
Trade-off Principle

Model source = trade-off between effort and flexibility. Pretrained models save the most time but offer the least customization. Custom Docker containers offer complete control but require the most engineering investment. Choose based on the problem's uniqueness and the team's capacity.

D2SageMaker Built-in Algorithm Categories

SageMaker provides optimized built-in algorithms across four task categories. Algorithm selection depends on two factors: the nature of the data (structured, image, text) and the task type (predict a label, group items, detect anomalies).

Supervised Learning
  • Classification — predict categorical labels
  • Regression — predict continuous values
  • Examples: XGBoost, Linear Learner, KNN
Unsupervised Learning
  • Clustering — group similar items
  • Dimensionality reduction — compress features
  • Anomaly detection — flag outliers
Image Processing
  • Image classification
  • Object detection
  • Computer vision tasks
Text Analysis
  • NLP tasks
  • Document classification
  • Machine translation
Key Points
  • Four source options: Pretrained (least effort) → Built-in → Custom Framework → Custom Docker (most control).
  • Four algorithm categories: Supervised · Unsupervised · Image Processing · Text Analysis.
  • JumpStart = primary access point for pretrained models within SageMaker. Faster development, reduced effort, ready infrastructure.
E
Evaluation Datasets and Model Fit
E
E1The Three Dataset Roles

A model's reported accuracy is only meaningful if it was measured on data the model has never seen. This is why the full dataset is split into three non-overlapping subsets, each serving a distinct purpose at a different stage of development.

Training Set
What the model learns from — the labeled examples used to fit model parameters. Typically 70–80% of available data.
Validation Set
Used iteratively during development — evaluates model performance after each training run, guides hyperparameter tuning, and helps choose between model architectures. The model can "see" this data indirectly through the tuning decisions it influences.
Test Set
The final, once-only evaluation. Held out completely until all development decisions are made. Provides an unbiased estimate of how the model will perform in production. Must remain untouched until final evaluation.
Why the Test Set Must Stay Sealed

If you peek at test set performance during development — even once — and use it to make a decision (select a model, adjust a feature), the test set is contaminated. It is no longer an unbiased estimate of production performance. This is called data leakage and is one of the most common mistakes in ML. [+]

E2Underfitting, Overfitting, and the Balanced Model

Model fit describes how well a trained model's predictions match reality — both on training data and, critically, on new unseen data. Three possible states:

Underfitting
  • Poor on training data
  • Poor on test data
  • Model too simple
  • High Bias, Low Variance
  • Fix: more complexity, better features
Overfitting
  • Great on training data
  • Poor on test data
  • Memorized noise
  • Low Bias, High Variance
  • Fix: regularization, more data, early stopping
Balanced ✓
  • Good on both
  • Generalizes to new data
  • Low Bias, Low Variance
  • Target state
Bias-Variance Analogy — Bullseye

Bias = how far the model's predictions are from the true center (wrong assumptions). Variance = how spread out predictions are across different samples (sensitivity to noise). High bias = consistently off-target (underfitting). High variance = scattered results that vary wildly (overfitting). Ideal = low bias AND low variance — tight cluster at the bullseye. [+]

Key Points
  • Training → learn · Validation → tune · Test → final unbiased evaluation (sealed until the end).
  • Underfitting: high bias, low variance — too simple. Overfitting: low bias, high variance — memorized noise.
  • Target: balanced model — low bias + low variance — generalizes well to unseen data.
  • Data leakage: using test set during development invalidates it as a fair performance estimate.

F2The Five Key Metrics

Each metric answers a different question about the model's behavior. The choice of which metric to optimize depends entirely on the business cost of each error type.

Accuracy
(TP+TN)/(TP+TN+FP+FN)
Proportion of correct predictions. Misleading with imbalanced data — a model predicting "not fraud" always is 99.9% accurate if fraud is 0.1% of cases.
Precision
TP / (TP + FP)
Of all predicted positives, how many were correct? Use when false positives are costly — e.g., spam filtering (flagging legitimate emails).
Recall
TP / (TP + FN)
Of all actual positives, how many did the model catch? Use when false negatives are critical — e.g., disease detection (missing a diagnosis).
F1 Score
Harmonic mean of P & R
Balances Precision and Recall. Use when both FP and FN matter and the data is imbalanced.
AUC-ROC
Area under ROC curve
Measures separability across all decision thresholds. AUC ≈ 0.5 = random guessing; AUC = 1.0 = perfect. Threshold-independent — useful for comparing models before choosing a decision cutoff.
Precision vs. Recall — The Core Trade-off

Increasing one typically decreases the other. Spam filter → optimize Precision (avoid blocking legitimate email). Cancer screening → optimize Recall (catch every positive case, even at cost of false alarms). The correct choice depends on which error type carries the higher business cost.

Key Points
  • Confusion matrix: TP (correct positive) · TN (correct negative) · FP (wrong alarm) · FN (missed positive).
  • Accuracy is misleading with imbalanced classes — don't rely on it alone.
  • Precision = minimize false positives. Recall = minimize false negatives.
  • F1 = balance both. AUC-ROC = threshold-independent comparison metric.
  • Metric choice = business cost of each error type.
G
Regression Metrics
G
G1MSE and R²

Regression models predict continuous numeric values (e.g., price, temperature, sales volume) rather than discrete categories. Two metrics dominate regression evaluation, each capturing a different dimension of model quality.

MSE — Mean Squared Error
  • Concept: average of (prediction − actual)² across all samples
  • Squaring penalizes large errors more than small ones
  • Lower MSE = better model — minimize toward zero
  • Has the same units as the target variable squared [+]
  • Sensitive to outliers — one large error inflates MSE significantly
R² — Coefficient of Determination
  • Concept: proportion of target variance explained by the model
  • Range: 0 to 1 (can be negative for very poor models) [+]
  • R² = 1.0 → model explains all variance (perfect fit)
  • R² = 0.0 → model performs no better than predicting the mean
  • Unitless — interpretable across different scales
How They Complement Each Other

MSE tells you how large the errors are. R² tells you how much of the variance the model explains. A model can have low MSE (small errors) but low R² (if the target barely varies). Use both together for a complete picture of regression model quality.

Key Points
  • MSE = average squared error — minimize toward zero. Penalizes outliers more heavily.
  • R² = explained variance — maximize toward 1. Unitless, interpretable across scales.
  • Use both together: MSE for error magnitude, R² for goodness of fit.
H
Business Metrics, Deployment, and Inference
H
H1Aligning ML Metrics with Business KPIs

A model with excellent technical metrics can still be a business failure if it optimizes the wrong thing. Business metrics measure real-world impact — sales increase, cost reduction, retention improvement — while ML metrics (accuracy, F1, MSE) measure model behavior in isolation.

The alignment challenge: a 1% improvement in model precision may translate to $2M in saved costs in fraud detection, while a 5% improvement in recall has no business value if the catch rate was already sufficient. [+] The connection between ML metrics and business outcomes must be explicitly modeled.

Error Cost Asymmetry
  • False positives → increased operational costs (manual reviews, customer friction)
  • False negatives → missed opportunities, safety risks, revenue loss
  • Cost function: assign dollar values to each error type
  • Optimize model threshold for business impact, not just metric score
Experimentation Methods
  • A/B Testing — two model versions deployed simultaneously; compare business metric outcomes
  • Canary Deployment — new model receives small % of traffic; monitor before full rollout
  • Both validate that technical gains translate to business gains
H2Deployment Types and Inference Options

Deploying a model means making it accessible for predictions in production. The fundamental choice is between self-hosted (you manage the infrastructure) and managed (AWS manages the infrastructure) deployments.

Self-Hosted API
  • Deploy on own infra: on-premise, cloud VMs, containers
  • Full control over environment and configuration
  • Potential cost savings at very high scale
  • High operational overhead — you manage scaling, patching, availability
Managed API (SageMaker)
  • One-click or API deployment to managed endpoints
  • Automatic scaling, secure HTTPS, multi-model hosting
  • No infrastructure management required
  • Less control vs self-hosted — AWS manages the infra layer

SageMaker supports four distinct inference types, each suited to different workload characteristics:

Real-time
Low-latency predictions via persistent endpoint. For interactive applications where users or systems expect near-instant responses. Always-on cost.
Batch Transform
Process entire large datasets at once — no persistent endpoint. Best for offline scoring jobs (e.g., overnight fraud scoring of all transactions). Lower cost, higher latency.
Asynchronous
Handles large payloads and long-running requests via a queue. Requests are submitted and results retrieved later. For video analysis, long document processing. [+]
Serverless
No infrastructure management, automatic scaling, cost-efficient for intermittent traffic. Cold start latency trade-off. Best when predictions are infrequent and unpredictable in volume.
Key Points
  • Best model = one that optimizes business KPIs, not just ML metrics. Align error costs to business impact.
  • A/B testing: compare models in production. Canary deployment: gradual rollout before full switch.
  • Self-hosted = more control, more overhead. Managed (SageMaker) = less control, faster deployment.
  • Four inference types: Real-time (low latency, persistent) · Batch (offline, no endpoint) · Async (large payloads, queued) · Serverless (intermittent, auto-scale).
I
MLOps — Principles, Pipelines, and AWS Services
I
I1What MLOps Is and Why It Exists

MLOps (Machine Learning Operations) is the discipline of operationalizing ML systems — combining people, processes, and technology to manage the full lifecycle from development through production monitoring. It is often described as DevOps for ML, but ML introduces challenges that traditional DevOps doesn't face: models degrade as real-world data changes, training is computationally expensive, and the "code" (model weights) cannot be reviewed like source code. [+]

Why Models Degrade
  • Real-world data distributions shift over time
  • Model trained on last year's data may not reflect this year's patterns
  • Data drift → feature distribution changes
  • Concept drift → the relationship between features and labels changes
MLOps Goals
  • Faster ML lifecycle delivery
  • Improved model quality and consistency
  • Strong collaboration between data scientists and engineers
  • Transparency, governance, and auditability
I2MLOps Benefits and Core Principles
Five Benefits
  • Productivity — faster experimentation, curated datasets
  • Reliability — CI/CD improves consistency and reduces human error
  • Repeatability — automated pipelines produce reproducible results
  • Auditability — track data, models, and experiments for compliance
  • Data & Model Quality — continuous bias and drift monitoring
Four Core Principles
  • Version Control — track code, data, and model versions for rollback
  • Automation — automate ingestion, training, testing, deployment
  • CI/CD for ML — continuous integration, delivery, training, monitoring
  • Model Governance — fairness, compliance, security, documentation
CI/CD in ML Has Two Extra Components

Traditional DevOps CI/CD: Continuous Integration (test code) + Continuous Delivery (deploy automatically). MLOps adds: Continuous Training (retrain models with new data automatically) and Continuous Monitoring (track model and business metrics in production). All four run together in a healthy MLOps system.

I3MLOps Pipelines and AWS Services

MLOps separates the ML workflow into two coordinated pipelines. The training pipeline produces validated model artifacts; the deployment pipeline takes approved artifacts and moves them through to production.

Training Pipeline
  • Data preparation → feature engineering
  • Model building → training run
  • Model evaluation → check performance criteria
  • Model registration → version and store if approved
  • Model approval → human or automated gate
Deployment Pipeline
  • Source → Build → Deploy → Test → Promote
  • Production monitoring — infrastructure, data quality, model performance
  • Drift detected → triggers retraining → loops back to training pipeline

AWS provides a full MLOps stack through SageMaker services:

SageMaker Pipelines
Automate end-to-end ML workflows — training, evaluation, registration, deployment
Model Registry
Version and manage model artifacts — approve or reject model versions before deployment
SageMaker Experiments
Track runs, hyperparameters, and metrics across training iterations for reproducibility
Model Monitor
Detect data drift and quality degradation in production — trigger retraining alerts
Feature Store
Centralized, versioned feature management for training and inference consistency
Data Wrangler
Low-code data prep that integrates directly into SageMaker Pipelines
Key Points
  • MLOps = DevOps for ML — people + processes + technology to operationalize the ML lifecycle.
  • Five benefits: Productivity · Reliability · Repeatability · Auditability · Data & Model Quality.
  • Four principles: Version Control · Automation · CI/CD · Governance.
  • CI/CD in ML = 4 components: Continuous Integration + Delivery + Training + Monitoring.
  • Two pipelines: Training (build → evaluate → register) · Deployment (build → deploy → test → promote → monitor).
  • AWS MLOps stack: Pipelines · Model Registry · Experiments · Model Monitor · Feature Store · Data Wrangler.
