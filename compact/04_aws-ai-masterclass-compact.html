Module Introduction

Module 4 covers how to develop, evaluate, deploy, and operate an ML system end-to-end — from translating business problems into ML tasks, through SageMaker tooling, evaluation metrics, and the operational discipline of MLOps.

A
The ML Development Lifecycle
A
A1The Seven Phases

ML development is an iterative loop — monitoring triggers retraining, which re-enters evaluation and re-deployment. It never terminates at deployment.

01 Business Goal
Define objective, success criteria, constraints (cost, latency, compliance)
02 Problem Framing
Translate into an ML task — classification, regression, clustering, etc.
03 Data Processing
Collection → Preprocessing (clean, transform) → Feature Engineering (extract signal)
04 Model Development
Training → Tuning (hyperparameters) → Evaluation (held-out data)
05 Deployment
Integrate trained model into production for real-world predictions
06 Monitoring
Track performance, detect drift, flag degradation in production
07 Retraining
Update model with new data when monitoring reveals performance drop
ML is an iterative loop — not a linear process that ends at deployment
B
Real-World ML Example — Call Center Routing
B
B1Business → ML Translation

Business problem: inefficient call routing — wrong agent, extra transfers, poor experience. ML task: predict correct agent skill before call connects = multiclass classification. Learning type: supervised (historical call records with known routing outcomes as labels).

Key features: recent orders · device ownership (Kindle) · membership status. Data prep: cleaning + label refinement (merge similar Kindle issue labels into one category). [+]

Data split
70–80% train · 10–15% validation · 10–15% test
Success metric
Business improvement (fewer transfers) — not model accuracy alone
Never evaluate on training data · Test set sealed until final evaluation

D
Model Sources and Built-in Algorithms
D
D1Four Source Options
Pretrained Models
  • Ready to deploy or fine-tune
  • Least effort — fastest time to value
  • JumpStart / Bedrock access
Built-in Algorithms
  • SageMaker-native, optimized for AWS
  • Moderate effort — configure, not code
  • Cover common supervised + unsupervised tasks
Custom Frameworks
  • TensorFlow, PyTorch, Scikit-learn
  • High flexibility — write your own model
  • Runs in SageMaker containers
Custom Docker
  • Fully custom environments
  • Maximum control — any language/library
  • Highest effort to build and maintain
Model source = effort vs flexibility trade-off. Pretrained = least effort · Docker = most control
D2Built-in Algorithm Categories
Supervised
  • Classification — categorical labels
  • Regression — continuous values
Unsupervised
  • Clustering · Dimensionality reduction
  • Anomaly detection
Image Processing
  • Image classification · Object detection
Text Analysis
  • NLP · Document classification · Translation
Algorithm selection = data type + task type
E
Evaluation Datasets and Model Fit
E
E1Three Dataset Roles
Training Set
What the model learns from — fits model parameters (~70–80%)
Validation Set
Guides tuning iteratively — evaluate after each run, choose hyperparameters (~10–15%)
Test Set
Final once-only evaluation — sealed until all development decisions are made (~10–15%)
Data leakage: using the test set during development contaminates it — no longer an unbiased production estimate. [+]
E2Model Fit States
Underfitting
  • Poor on train AND test
  • High Bias · Low Variance
  • Model too simple
Overfitting
  • Great train · Poor test
  • Low Bias · High Variance
  • Memorized noise
Balanced ✓
  • Good on both
  • Low Bias · Low Variance
  • Target state
Bias = off-target (underfitting)Variance = scattered (overfitting)Target: low bias + low variance
F
Classification Metrics
F
F1Confusion Matrix
Predicted PositivePredicted Negative
Actual PositiveTP — Correct positiveFN — Missed positive
Actual NegativeFP — Wrong alarmTN — Correct negative
F2The Five Metrics
Accuracy
(TP+TN)/Total
Misleading with imbalanced data
Precision
TP/(TP+FP)
Minimize false positives — use for spam filtering
Recall
TP/(TP+FN)
Minimize false negatives — use for disease detection
F1 Score
Harmonic mean P & R
Balance both when FP and FN both matter
AUC-ROC
Area under ROC curve
Threshold-independent. AUC≈0.5=random · AUC=1=perfect
Precision = minimize FP (spam)Recall = minimize FN (disease)Metric choice = business cost of each error type
G
Regression Metrics
G
G1MSE and R²
MSE
Mean Squared Error — average of (prediction − actual)². Lower = better. Penalizes large errors. Minimize toward zero.
R²
Coefficient of Determination — proportion of variance explained. Range 0–1. Closer to 1 = better fit. Unitless.
MSE = error magnitudeR² = goodness of fitUse both together for complete picture
H
Business Metrics, Deployment, and Inference
H
H1Business KPI Alignment

Technical metrics (accuracy, F1) measure model behavior. Business metrics (sales, cost, retention) measure real-world impact. A model with excellent ML metrics can still fail if it optimizes the wrong thing.

Error Cost
FP → operational cost (reviews, friction) · FN → missed revenue, safety risk
Cost Function
Assign dollar values to each error type — optimize threshold for business impact
A/B Testing
Two model versions live simultaneously — compare real business metric outcomes
Canary Deployment
New model receives small % traffic first — validate before full rollout
H2Deployment Types and Inference Options
Self-Hosted API
  • Own infra (on-prem, cloud VMs)
  • Full control + customization
  • High operational overhead
Managed API (SageMaker)
  • Auto scaling, HTTPS, multi-model
  • No infra management
  • Less control vs self-hosted
Real-time
Low latency, persistent endpoint — interactive apps, always-on cost
Batch Transform
Large dataset, no persistent endpoint — offline scoring, lower cost
Asynchronous
Large payloads, queued requests — video analysis, long docs [+]
Serverless
Auto-scale, no infra — intermittent traffic, cold start trade-off
Self-hosted = control · Managed = speedInference type = latency + workload requirements
I
MLOps — Principles, Pipelines, AWS Stack
I
I1What MLOps Is

MLOps = DevOps for ML — people + processes + technology to operationalize the ML lifecycle. ML models degrade as data distributions shift (data drift) or label relationships change (concept drift). MLOps ensures they stay useful over time. [+]

5 Benefits
  • Productivity — faster experimentation
  • Reliability — CI/CD consistency
  • Repeatability — automated pipelines
  • Auditability — track data/models/experiments
  • Data & Model Quality — bias + drift monitoring
4 Principles
  • Version Control — code, data, models
  • Automation — ingestion to deployment
  • CI/CD — 4 components (below)
  • Governance — fairness, compliance, docs
CI/CD in ML = 4 components: Continuous Integration (test) + Continuous Delivery (deploy) + Continuous Training (retrain) + Continuous Monitoring (track). Traditional DevOps only has the first two.
I2Pipelines and AWS MLOps Stack
Training Pipeline
  • Data prep → Feature engineering
  • Model build → Training run
  • Evaluation → Registration (if approved)
  • Model approval gate
Deployment Pipeline
  • Source → Build → Deploy → Test → Promote
  • Production monitoring
  • Drift → triggers retraining → loops back
SageMaker Pipelines
Automate end-to-end ML workflows
Model Registry
Version, approve/reject model artifacts
SageMaker Experiments
Track runs, hyperparameters, metrics
Model Monitor
Detect drift, trigger retraining alerts
Feature Store
Versioned features for training + inference
Data Wrangler
Low-code prep integrated into Pipelines
MLOps = DevOps + Continuous Training + Continuous MonitoringTwo pipelines: Training (build/evaluate/register) · Deployment (deploy/test/promote/monitor)
Module Summary
  • ML lifecycle = 7-phase iterative loop ending at retraining, not deployment. Data Processing = Collection → Preprocessing → Feature Engineering.
  • Business → ML: call center routing = multiclass classification + supervised learning. Split: 70–80/10–15/10–15. Test set sealed. Success = business improvement.
  • SageMaker tools: Data Wrangler · Feature Store · Canvas (no-code) · JumpStart (pretrained) · Experiments · Hyperparameter Tuning · Model Monitor. Artifacts in S3.
  • Model sources: Pretrained (least effort) → Built-in → Custom Framework → Custom Docker (most control). Algorithm categories: Supervised · Unsupervised · Image · Text.
  • Fit: Underfitting (high bias) · Overfitting (high variance) · Balanced (both low). Metrics: Accuracy (imbalanced limitation) · Precision (FP cost) · Recall (FN cost) · F1 (balance) · AUC-ROC (threshold-independent). MSE (error) + R² (fit). Inference: Real-time · Batch · Async · Serverless.
  • MLOps = DevOps for ML. 5 benefits · 4 principles · 4 CI/CD components (Integration, Delivery, Training, Monitoring). Two pipelines. AWS stack: Pipelines · Registry · Experiments · Monitor · Feature Store.
